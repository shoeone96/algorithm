# 디버깅 순서: 설계 의심 전에 구현 대조부터

> **문제**: Generate Parentheses (LeetCode 22)
> **유형**: 백트래킹
> **코드**: `GenerateParentheses.java`

---

## 상황

- 상태 트리를 그려서 설계한 뒤 코드 작성
- 결과가 틀려서 "내 전제(설계)가 틀렸나?" 방향으로 고민
- 실제 원인: `)` 추가 시 `rightCount - 1` 해야 하는데 `leftCount - 1`로 잘못 씀

---

## 깨달음

**버그가 나면 설계를 의심하기 전에, 코드가 설계를 정확히 반영했는지 먼저 대조하자.**

### 디버깅 순서

1. **구현 대조**: 내가 쓴 코드가 내 설계(상태 트리)를 정확히 반영했나?
2. **손 트레이싱**: 작은 케이스(n=2)로 코드 기준으로 흐름 따라가기
3. **설계 재검토**: 위 두 단계에서 안 잡히면 그때 전제를 의심

### 왜 이 순서인가?

- "전제가 틀렸나?"로 가면 설계를 다시 의심하게 되고, 실제로는 구현 실수인데 멀리 돌아감
- 설계 단계에서 상태 트리를 이미 그렸다면, 설계는 맞을 확률이 높음
- 대부분의 버그는 **설계 → 코드 변환 과정**에서 발생 (변수명 실수, 인덱스 오류 등)

---

## 손 트레이싱 타이밍

| 타이밍 | 목적 |
|--------|------|
| 설계 단계 | 접근법이 맞는지 검증 (상태 트리) |
| **코드 작성 후** | 코드가 설계를 정확히 구현했는지 검증 |

설계 때만 상태 트리를 그리고 코드 작성 후에는 안 하면, 이번 같은 구현 실수를 놓친다.
